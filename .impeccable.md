# .impeccable — ESAP-KB Admin design context

This file is the locked design context for the ESAP-KB Admin frontend. It is the
source of truth for palette, typography, motion, and aesthetic constraints.
Any UI work must respect these without deviation.

## Product

- **Name:** ESAP-KB Admin
- **What it is:** Enterprise knowledge-base admin dashboard. Customers manage
  companies, users, database connections, roles, and chat with their data
  through an AI assistant.
- **Who uses it:** External enterprise customers. This is their primary admin
  tool — they log in daily.
- **Tone:** Professional enterprise. Calm, trustworthy, slightly formal.
- **Primary emotional goal:** *Trust.* The interface should feel reliable,
  authoritative, and incapable of surprising the user in a bad way. Every
  pixel earns credibility.
- **Secondary feeling:** *I know exactly where I am.* Spatial clarity,
  wayfinding, and orientation reinforce the trust goal.
- **Design lineage:** Supabase-inspired dark professional. A dashboard dark,
  not a marketing-site dark. Precision-tool aesthetic, not neon-studio.

## Design Context

### Users
External enterprise customers — admins and technical leads. They log in daily
to manage companies, users, database connections, roles, and to run AI-assisted
queries against their data. Context of use: work hours, office or home office,
focused task completion. They are not exploring; they are working.

### Brand personality
**Precise · Reliable · Quiet**. The product should feel like a well-calibrated
instrument, not a consumer app. No delight for delight's sake; every interaction
should reduce cognitive load and increase confidence.

### Aesthetic direction
Supabase dashboard is the north star. Dark-only, always. The design must earn
trust through consistency, restraint, and visual stability — not through
ornamentation. Surfaces are dark and quiet; the green accent appears only where
it earns attention (active states, primary CTAs, positive signals).

### Design principles
1. **Trust through consistency.** Identical elements behave identically. No
   surprises, no context-dependent quirks. Predictability = trust.
2. **Hierarchy by contrast, not decoration.** Size and weight do the work.
   Color is reserved for state changes and semantic meaning.
3. **Density without clutter.** Information-rich layouts with generous vertical
   rhythm. Tables and lists over card grids.
4. **Empty states teach, not apologize.** Every zero-state explains the next
   action. Never just "No data found."
5. **Motion earns its place.** Only entrance/exit transitions and state
   feedback. 220ms max. No ambient animation.

### Known areas needing improvement (audit targets)
- Login / auth screen — needs to feel like the same product as the dashboard
- Dashboard home — lacks a clear hierarchy and purpose statement
- All pages generally — consistency pass needed (spacing, heading sizes, empty
  states, button hierarchy)

### Non-negotiables
- Dark-only — no light mode, ever
- Geist / Geist Mono type stack
- Supabase green (#3ecf8e) as the sole accent
- No glowing effects, no gradient text, no nested cards

## Aesthetic rules (non-negotiable)

### Don't
- No glowing accents, no neon, no blooming shadows.
- No AI-slop palette: cyan gradients, purple-to-blue, neon on dark.
- No gradient text, anywhere, ever.
- No large icons with rounded corners stacked above every heading.
- No wrapping every element in a card; no nested cards.
- No identical card grids repeating the same icon/heading/text block.
- No modal when an inline pattern would do.
- No monospace as a lazy shorthand for "technical vibes".
- Never animate layout properties (width, height, padding). Transform +
  opacity only.
- No bounce or elastic easing.
- No pure black (#000) or pure white (#fff).

### Do
- Modern CSS only — oklch, color-mix, CSS custom properties.
- Tint all neutrals toward the brand hue (green-tinted darks).
- Varied spacing to create rhythm — tight clusters, generous breaks.
- Exponential easing (`cubic-bezier(0.16, 1, 0.3, 1)`) for motion.
- Empty states teach the interface; they don't say "nothing here".
- Progressive disclosure — start simple, reveal on interaction.

## Typography — two-voice system

- **Geist** — all UI (headings, labels, body, buttons).
- **Geist Mono** — timestamps, IDs, DB names, technical tokens ONLY. Used
  sparingly; never decorative.

Instrument Serif is removed from the stack. `font-serif` and
`font-feat-cswh` utilities are deprecated and must not be used.

## Motion tokens

```css
@keyframes fade-up { from { opacity: 0; transform: translateY(6px) } to { opacity: 1; transform: translateY(0) } }
@keyframes fade-in { from { opacity: 0 } to { opacity: 1 } }
@keyframes shimmer { from { background-position: 200% 0 } to { background-position: -200% 0 } }
```

Utilities: `animate-fade-up`, `animate-fade-in`,
`animation-delay-{50,100,150,200,300}`.

Entrance duration: 220ms with `cubic-bezier(0.16, 1, 0.3, 1)`. Stagger delays
cap at 300ms total. Respect `prefers-reduced-motion`.

## Focus ring

Keyboard focus uses the accent green at 40% opacity:
`box-shadow: 0 0 0 2px rgba(62,207,142,0.4)`.

## Review checklist

Before shipping any UI change:

1. No warm-cream hex values remain (`#f2f1ed`, `#26251e`, `#f54e00`,
   `#ebeae5`, etc.).
2. No `font-serif` or `font-feat-cswh` classes in app code.
3. No `Instrument_Serif` anywhere.
4. Contrast meets WCAG AA (4.5:1 body, 3:1 UI).
5. Run the AI-slop test: would someone instantly say "AI made this"? If yes,
   fix it.
